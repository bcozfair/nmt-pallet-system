3.4.2.1  Use Case เข้าสู่ระบบ (Login)
Use Case Description
Use Case Title: เข้าสู่ระบบ (Login)	Use Case ID: UC-001
Primary Actor: ผู้ดูแลระบบ (Admin), เจ้าหน้าที่ (Staff)
Secondary Actor: -
Description: ผู้ใช้งานเข้าสู่ระบบเพื่อยืนยันตัวตนและรับสิทธิ์การใช้งานตามบทบาทของตนเอง
Pre-conditions:
1.	ผู้ใช้งานมีบัญชีในระบบแล้ว
Post-conditions:
1.	ระบบสร้าง Session สำหรับสำหรับผู้ใช้งาน
2.	ผู้ใช้งานถูกนำไปยังหน้าหลักตามสิทธิ์ผู้ใช้งาน
Main Flow:
1.	ผู้ใช้งานเข้าสู่หน้า Login
2.	ระบุ ชื่อผู้ใช้และรหัสผ่าน
3.	กดปุ่ม Sign In
4.	ระบบตรวจสอบความถูกต้องของข้อมูล
5.	ระบบตรวจสอบสิทธิ์ (Role) ของผู้ใช้งาน
6.	ระบบนำผู้ใช้งานไปยังหน้าจอหลัก
6.1 กรณีเป็น Admin นำไปยังหน้า Admin Dashboard
6.2 กรณีเป็น Staff นำไปยังหน้า Mobile Home
Alternative Flows:
1.	 ผู้ใช้งานสามารถกดแสดงรหัสผ่าน เพื่อดูรหัสผ่านที่กรอกได้
Exception Flows:
1.	หากรหัสผ่านไม่ถูกต้อง ระบบจะแจ้งเตือนความผิดพลาดให้ผู้ใช้กรอกรหัสผ่านใหม่อีกครั้ง
2.	หากบัญชีถูกระงับการใช้งาน ระบบจะแจ้งเตือนความผิดพลาดและไม่สามารถเข้าสู่ระบบได้

3.4.2.2  Use Case บันทึกรายการเบิกพาเลท (Check-out Pallet)
Use Case Description
Use Case Title: บันทึกรายการเบิกพาเลท (Check-out Pallet)	Use Case ID: UC-002
Primary Actor: เจ้าหน้าที่ (Staff)
Secondary Actor: -
Description: เจ้าหน้าที่ดำเนินการเบิกพาเลทออกจากพื้นที่จัดเก็บเพื่อนำไปใช้งานยังแผนกต่างๆ
Pre-conditions:
1.	เจ้าหน้าที่ล็อกอินเข้าสู่ระบบเรียบร้อยแล้ว
Post-conditions:
1.	สถานะของพาเลทในฐานข้อมูลเปลี่ยนเป็น "In Use"
2.	ตำแหน่งปัจจุบัน (Current Location) ของพาเลทถูกอัปเดตเป็นแผนกปลายทางที่ระบุ
3.	สร้างรายการธุรกรรม (Transaction) ประเภท "Check-out" บันทึกในประวัติ
Main Flow:
1.	เจ้าหน้าที่เลือกเมนู Check Out จากหน้าหลัก
2.	ระบบแสดงหน้าจอเลือกแผนกปลายทาง
3.	เจ้าหน้าที่เลือกแผนกปลายทางที่ต้องการ
4.	ระบบเปิดกล้องและเข้าสู่โหมดสแกน QR Code
5.	เจ้าหน้าที่สแกน QR Code ที่ติดอยู่บนพาเลท
6.	ระบบตรวจสอบข้อมูลพาเลทในฐานข้อมูล
6.1	ตรวจสอบว่าพบรหัสพาเลทในระบบ
6.2	ตรวจสอบว่าสถานะพาเลทไม่เป็น “Damaged”
7.	ระบบเพิ่มพาเลทลงในรายการรอยืนยัน (Pending List)  
8.	เจ้าหน้าที่สามารถสแกนพาเลทชิ้นถัดไปได้ทันที (Batch Scan)
9.	เจ้าหน้าที่กดปุ่มยืนยัน Check Out  
10.	ระบบบันทึกรายการเบิกพาเลททั้งหมดและกลับสู่หน้าหลัก
Alternative Flows:
1.	สามารถเลือกลบบางรายการออกจาก Pending List ก่อนยืนยันได้
Exception Flows:  
1.	เมื่อสแกนไม่พบข้อมูลพาเลท ระบบจะแจ้งเตือนความผิดพลาดและไม่เพิ่มรายการดังกล่าว
2.	เมื่อสแกนพาเลทชำรุด ระบบจะแจ้งเตือนความผิดพลาดและไม่อนุญาตให้เบิกพาเลทดังกล่าว
3.	การสแกนซ้ำ ระบบจะแจ้งเตือนความผิดพลาดและไม่เพิ่มรายการซ้ำ
4.	การบันทึกล้มเหลว ระบบจะแจ้งเตือนความผิดพลาดและคงรายการไว้ให้สามารถยืนยันใหม่ได้

Main Flow:
1.	เจ้าหน้าที่เลือกเมนู Check In จากหน้าหลัก

3.	เจ้าหน้าที่สแกน QR Code ที่ติดอยู่บนพาเลท
4.	ระบบตรวจสอบข้อมูลพาเลทในฐานข้อมูล
4.1	ตรวจสอบว่าพบรหัสพาเลทในระบบ
4.2	ตรวจสอบว่าสถานะพาเลทไม่เป็น “Damaged”
5.	ระบบเพิ่มพาเลทลงในรายการรอยืนยัน (Pending List)
6.	เจ้าหน้าที่สามารถสแกนพาเลทชิ้นถัดไปได้ทันที (Batch Scan)
7.	เจ้าหน้าที่กดปุ่ม ยืนยัน Check In
8.	ระบบบันทึกรายการคืนพาเลททั้งหมดและกลับสู่หน้าหลัก
Alternative Flows:
1.	สามารถเลือกลบบางรายการออกจาก Pending List ก่อนยืนยันได้
2.	สามารถยกเลิกการทำงานได้ ระบบล้างรายการที่สแกนทั้งหมดและกลับสู่หน้าหลัก
Exception Flows:  
5.	เมื่อสแกนไม่พบข้อมูลพาเลท ระบบจะแจ้งเตือนความผิดพลาดและไม่เพิ่มรายการดังกล่าว
6.	เมื่อสแกนพาเลทชำรุด ระบบจะแจ้งเตือนความผิดพลาดและไม่อนุญาตให้เบิกพาเลทดังกล่าว
7.	การสแกนซ้ำ ระบบจะแจ้งเตือนความผิดพลาดและไม่เพิ่มรายการซ้ำ
8.	การบันทึกล้มเหลว ระบบจะแจ้งเตือนความผิดพลาดและคงรายการไว้ให้สามารถยืนยันใหม่ได้

3.4.2.4	Use Case บันทึกรายการแจ้งพาเลทชำรุด (Report Damage)
Use Case Description
Use Case Title: บันทึกรายการแจ้งพาเลทชำรุด (Report Damage)	Use Case ID: UC-004
Primary Actor: เจ้าหน้าที่ (Staff)
Secondary Actor: -
Description: เจ้าหน้าที่รายงานพาเลทที่เสียหายพร้อมแนบรูปภาพหลักฐาน
Pre-conditions:
1.	พบพาเลทที่มีความเสียหายทางกายภาพ
Post-conditions:
1.	สถานะพาเลทเปลี่ยนเป็น "Damaged"
2.	รูปภาพหลักฐานถูกบันทึกในระบบ
3.	สร้างรายการธุรกรรม (Transaction) ประเภท "Report Damage"
Main Flow:
1.	เจ้าหน้าที่เลือกเมนู Report Damage จากหน้าหลัก
2.	เจ้าหน้าที่สแกน QR Code ของพาเลทที่ชำรุด
3.	ระบบตรวจสอบสถานะพาเลท (ต้องยังไม่ถูกแจ้งชำรุดมาก่อน)
4.	ระบบแสดงหน้าจอแบบฟอร์มแจ้งชำรุด (Damage Form)
5.	เจ้าหน้าที่กดปุ่ม Upload Evidence เพื่อถ่ายรูปยืนยันสภาพความเสียหาย
6.	ระบบแสดงตัวอย่างรูปภาพที่อัปโหลด
7.	เจ้าหน้าที่กดปุ่ม Submit Report
8.	ระบบบันทึกสถานะพาเลทและรูปภาพ จากนั้นกลับสู่หน้าหลัก
Alternative Flows:
1.	หาก QR Code ชำรุดอ่านค่าไม่ได้ สามารถพิมพ์รหัสพาเลทแทนการสแกนได้
Exception Flows:  
1.	เมื่อสแกนพาเลทมีสถานะชำรุดอยู่แล้ว ระบบจะแสดงข้อความแจ้งเตือนและไม่เปิดหน้าฟอร์ม
2.	ถ้าไม่ได้แนบรูปหลักฐาน ปุ่ม Submit Report จะไม่ทำงาน (Disable) จนกว่าจะมีรูปภาพ

3.4.2.5	Use Case ตรวจสอบประวัติการทำรายการ (View History)
Use Case Description
Use Case Title: ตรวจสอบประวัติการทำรายการ (View History)	Use Case ID: UC-005
Primary Actor: เจ้าหน้าที่ (Staff)
Secondary Actor: -
Description: เจ้าหน้าที่ดูรายการธุรกรรมล่าสุดที่ตนเองเป็นผู้ดำเนินการ
Pre-conditions:
1.	เจ้าหน้าที่ล็อกอินเข้าสู่ระบบเรียบร้อยแล้ว
Post-conditions:
1.	เจ้าหน้าที่รับทราบข้อมูลประวัติการทำรายการย้อนหลัง และข้อมูลตามเงื่อนไขที่ค้นหา
Main Flow:
1.	เจ้าหน้าที่เลือกเมนู History
2.	ระบบแสดงรายการประวัติการทำรายการล่าสุด (Default View)
3.	กรณีค้นหาข้อมูล
3.1	พิมพ์คำค้นหา เช่น รหัสพาเลท หรือชื่อสถานที่
3.2	ระบบแสดงรายการที่ตรงกับคำค้นหา
4.	กรณีกรองข้อมูล
4.1  เจ้าหน้าที่กดปุ่ม Filter
4.2  เลือกเงื่อนไขที่ต้องการ ได้แก่ ช่วงวันเวลา และประเภทรายการ (เบิก-คืน/แจ้งชำรุด)
5.	ระบบแสดงรายการที่ตรงตามเงื่อนไข พร้อมรายละเอียด ได้แก่ รหัสพาเลท วันที่และเวลา (Timestamp) และประเภทรายการ
Alternative Flows:
1.	กรณีไม่มีข้อมูล ระบบแสดงข้อความแจ้งว่าไม่มีประวัติการทำรายการ
Exception Flows:
1.	หากระบบขัดข้อง ไม่สามารถดึงข้อมูลประวัติได้ ระบบจะแจ้งเตือนความผิดพลาด

3.4.2.6	Use Case จัดการข้อมูลพาเลท (Manage Inventory)
Use Case Description
Use Case Title: จัดการข้อมูลพาเลท (Manage Inventory)	Use Case ID: UC-006
Primary Actor: ผู้ดูแลระบบ (Admin)
Secondary Actor: -
Description: บริหารจัดการข้อมูลพาเลท ทั้งการเพิ่ม แก้ไข ลบ ค้นหา สร้างและพิมพ์ QR Code
Pre-conditions:
1.	เข้าสู่ระบบด้วยสิทธิ์ Admin เรียบร้อยแล้ว
Post-conditions:
1.	ข้อมูลพาเลทในระบบถูกเพิ่ม แก้ไข หรือลบ ตามคำสั่ง
Main Flow:
1.	Admin เลือกเมนู Inventory
2.	ระบบแสดงตารางรายการพาเลททั้งหมด
3.	กรณีเพิ่มพาเลท
3.1	 กดปุ่ม Add Palette
3.2	 ระบุ ID และสถานที่เริ่มต้น แล้วกด Create เพื่อบันทึกการเพิ่มรายการ
4.	กรณีค้นหาหรือกรอง
4.1	 พิมพ์รหัสพาเลทเพื่อค้นหาข้อมูลพาเลท
4.2	 เลือกกรองตามเงื่อนไข เช่น สถานะ วันเวลา หรือสถานที่ เพื่อกรองข้อมูลพาเลท
4.3	 ระบบแสดงผลลัพธ์ตามการค้นหาหรือการกรอง
5.	กรณีสร้างหรือพิมพ์ QR Code
5.1	 เลือกรายการพาเลทที่ต้องการ แล้วกดปุ่ม Print QR
6.	กรณีสร้างรายการเบิก-คืน (Admin Transaction)
6.1	 เลือกรายการพาเลท แล้วกดปุ่ม Create Transaction
6.2	 ระบุประเภท (Check In / Check Out) สถานที่ปลายทาง และวันเวลาที่ต้องการ
6.3	 กดปุ่ม Save เพื่อบันทึกรายการ
7.	กรณีลบพาเลท
7.1	เลือกรายการพาเลทแล้วกดปุ่ม Delete
7.2	กดยืนยันการลบที่หน้าต่าง Confirm Dialog
8.	กรณีแก้ไขข้อมูลพาเลท
8.1	กดปุ่ม Edit ที่รายการพาเลท
8.2	แก้ไขรายละเอียด (เช่น Remark) แล้วกดปุ่ม Save เพื่อบันทึกการเปลี่ยนแปลง
Alternative Flows: -
Exception Flows:
1.	หากเพิ่มรหัสพาเลทซ้ำ ระบบจะแจ้งเตือนความผิดพลาดและไม่เพิ่มรายการซ้ำ
2.	หากลบพาเลทที่ถูกใช้งานอยู่ ระบบจะแจ้งเตือนความผิดพลาดและไม่สามารถลบได้

3.4.2.7	Use Case จัดการข้อมูลผู้ใช้งาน (Manage Users)
Use Case Description
Use Case Title: จัดการข้อมูลผู้ใช้งาน (Manage Users)	Use Case ID: UC-007
Primary Actor: ผู้ดูแลระบบ (Admin)
Secondary Actor: -
Description: บริหารจัดการบัญชีผู้ใช้งานและสิทธิ์การเข้าถึง
Pre-conditions:
1.	เข้าสู่ระบบด้วยสิทธิ์ Admin เรียบร้อยแล้ว
Post-conditions:
1.	ฐานข้อมูลผู้ใช้งานได้รับการอัปเดต (สร้างใหม่/แก้ไข/ลบ/เปลี่ยนสิทธิ์)
Main Flow:
1.	Admin เลือกเมนู User Management
2.	กรณีเพิ่มผู้ใช้
2.1	กดปุ่ม Create User
2.2	ระบุข้อมูลพนักงาน, แผนก, และกำหนด Role (Staff/Admin)
2.3	กดปุ่ม Save เพื่อบันทึกข้อมูล
3.	กรณีค้นหาหรือกรอง
3.1	พิมพ์ข้อมูลเพื่อค้าหา เช่น ชื่อหรือรหัสพนักงาน
3.2	ระบุเงื่อนไขการกรอง เช่น กรองตามแผนก
3.3	ระบบแสดงผลลัพธ์ตามการค้นหาหรือการกรอง
4.	กรณีแก้ไขข้อมูลหรือปรับเปลี่ยนสิทธิ์
4.1	กดปุ่ม Edit ที่รายชื่อผู้ใช้งาน
4.2	แก้ไขข้อมูล หรือเปลี่ยนสิทธิ์ Role (Staff/Admin)
4.3	กดปุ่ม Save เพื่อบันทึกข้อมูล
5.	กรณีรีเซ็ตรหัสผ่าน
5.1	กดเมนู Reset Password ที่รายชื่อผู้ใช้
5.2	ระบุรหัสผ่านใหม่
5.3	กดปุ่ม Save เพื่อบันทึกข้อมูล
Alternative Flows:
1.	สามารถกดปุ่ม เปิด-ปิด สถานะผู้ใช้งาน (active/Inactive) เพื่อระงับการใช้งานชั่วคราวได้
Exception Flows:

3.	กรณีเพิ่มแผนกหรือสถานที่
3.1	กดปุ่ม Add Location
3.2	ระบุชื่อแผนกหรือสถานที่
3.3	กดปุ่ม Save เพื่อบันทึกข้อมูล

4.2	กดปุ่ม Save เพื่อบันทึกข้อมูล
5.	กรณีค้นหาหรือกรองข้อมูล

6.1	กดปุ่ม Delete ที่รายชื่อแผนกหรือสถานที่
6.2	กดยืนยันการลบที่หน้าต่าง Confirm Dialog
Alternative Flows:
1.	สามารถกดปุ่ม เปิด-ปิด สถานะการทำงาน (active/Inactive) เพื่อระงับการใช้งานชั่วคราวได้
Exception Flows:
1.	หากลบแผนกหรือสถานที่ที่มีพาเลทอยู่ ระบบจะแจ้งเตือนความผิดพลาดและไม่สามารถลบได้

Use Case Description
Use Case Title: จัดการข้อมูลประวัติการทำรายการ (Manage Transaction)	Use Case ID: UC-009
Primary Actor: ผู้ดูแลระบบ (Admin)
Secondary Actor: -

4.	กรณีแก้ไขประวัติ
4.1	กดปุ่ม Edit ที่รายการ
4.2	แก้ไขข้อมูล เช่น เปลี่ยนแผนกปลายทาง

5.	กรณีลบประวัติ
5.1	กดปุ่ม Delete ที่รายการธุรกรรม
5.2 กดยืนยันการลบที่หน้าต่าง Confirm Dialog
Alternative Flows:
1.	กรณีเป็นรายการแจ้งชำรุด Report Damage สามารถคลิกดูรูปภาพหลักฐานที่แนบมาได้

1.	หากไม่พบรายการที่ค้นหา ระบบจะแจ้งเตือนความผิดพลาด

3.4.2.12  Use Case ลืมรหัสผ่าน (Forget Password)
Use Case Description
Use Case Title: ลืมรหัสผ่าน (Forget Password)	Use Case ID: UC-012
Primary Actor: ผู้ดูแลระบบ (Admin)
Secondary Actor: -
Description: ผู้ดูแลระบบทำการขอรีเซ็ตรหัสผ่านใหม่ทางอีเมล กรณีจำรหัสผ่านเดิมไม่ได้
Pre-conditions:
1.	ผู้ดูแลระบบต้องมีบัญชีอีเมลที่ลงทะเบียนไว้ในระบบ
Post-conditions:
1.	ระบบส่งลิงก์สำหรับตั้งค่ารหัสผ่านใหม่ไปยังอีเมลของผู้ดูแลระบบ
Main Flow:
1.	ผู้ดูแลระบบกดปุ่ม ลืมรหัสผ่าน (Forget Password) ที่หน้า Login
2.	ระบบแสดงหน้าจอให้เลือกรายชื่อบัญชีผู้ดูแลระบบ
3.	กดปุ่ม รีเซ็ตรหัสผ่าน (Reset Password)
4.	ระบบส่งอีเมลที่มีลิงก์สำหรับรีเซ็ตรหัสผ่านไปยังอีเมลกลางของระบบ
5.	ระบบแจ้งเตือนให้ตรวจสอบกล่องข้อความในอีเมล
Alternative Flows:
1.	ผู้ดูแลระบบกดปุ่ม Back to Login เพื่อกลับไปหน้าเข้าสู่ระบบ
Exception Flows:
1.	หากอีเมลไม่ถูกต้องหรือไม่มีในระบบ ระบบจะแจ้งเตือนความผิดพลาดและไม่ส่งอีเมล
2.	หากระบบส่งอีเมลล้มเหลว ระบบจะแจ้งเตือนความผิดพลาดและให้ลองใหม่อีกครั้ง

3.4.2.13  Use Case แจ้งเตือนอัตโนมัติผ่าน LINE (Automatic Line Notification)
Use Case Description
Use Case Title: แจ้งเตือนอัตโนมัติผ่าน LINE (Automatic Line Notification)	Use Case ID: UC-013
Primary Actor: ตัวตั้งเวลา (Scheduler/Cron)
Secondary Actor: LINE Messaging API
Description: ระบบทำการส่งสรุปข้อมูลรายงานประจำวัน (Daily Report) ไปยังแอปพลิเคชัน LINE ตาม LINE Group หรือ LINE ID ที่ตั้งค่าไว้
Pre-conditions:
1.	ตั้งค่า Line Channel Access Token และ Target ID เรียบร้อยแล้ว
2.	ถึงเวลาที่กำหนดส่งรายงาน (Scheduled Time)
Post-conditions:
1.	ผู้ใช้ได้รับข้อความแจ้งเตือนผ่านแอปพลิเคชัน LINE
Main Flow:
1.	ระบบตรวจสอบเวลาปัจจุบันตรงกับรอบส่งรายงาน
2.	ระบบรวบรวมข้อมูลสรุป (จำนวนพาเลทเข้า-ออก, คงเหลือ, เกินกำหนด)
3.	ระบบสร้างข้อความในรูปแบบ Flex Message
4.	ระบบส่งข้อความไปยังแอปพลิเคชัน LINE ผ่าน LINE Messaging API
5.	LINE Messaging API ตอบกลับสถานะสำเร็จ
6.	ผู้ใช้ได้รับข้อความแจ้งเตือน และการแสดงผลข้อมูลผ่านแอปพลิเคชัน LINE
Alternative Flows: -
Exception Flows:
1.	หากส่งไม่สำเร็จระบบจะบันทึก Log ความผิดพลาดและพยายามส่งใหม่ในรอบถัดไป
